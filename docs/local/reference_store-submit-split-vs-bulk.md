# [参考] ストア申請: 分割 vs 一括 — 審査日数と離脱リスク

作成日: 2026-07-19(日)

スターターパック取込（metrics 付き固定 URL fetch）を v0.1 申請に含めるか、  
「URL のみ第1弾 → 後からスターター」に分けるかの判断記録。  
日数目安は保証ではない（CWS / AMO の運用で振れる）。

## 1. 結論（既定）

**一括申請（スターター本線込み）を既定とする。**

- 分割で稼げる審査日数は、だいたい **数日〜長くて 2 週間前後**（差し戻しなし想定）
- 中途半端な第1弾（空 + teach 必須・数字まで遠い）の **離脱・低評価の方が痛い**
- 審査を遅くしがちな本体は「再取得ボタン」や「アイコン同梱」ではなく、  
  **metrics 付きリモート設定 + privacy/listing 説明不足**
- 権限を増やさず（既存 `raw.githubusercontent.com/ishizakahiroshi/*`）、  
  data only・確認後のみ GET・eval しない・初回自動 fetch なし、を文面と一致させれば一括で戦える

関連 plan の D8（UX をストア承認より優先）と同じ向き。

## 2. 審査日数の目安

| 状況 | Chrome Web Store（目安） | Firefox AMO（目安） |
|------|--------------------------|---------------------|
| 初回・素直な拡張 | 数日〜2週間 | 数日〜2週間（人手寄りで振れやすい） |
| 更新（小さな差分） | 数時間〜数日 | 数日 |
| privacy / 権限 / リモート設定説明が絡む | +数日〜差し戻し1往復 | 同様 |

分割 vs 一括で **自動的に +1 ヶ月** になる類の差ではない。  
差が出やすいのは:

1. **差し戻し 1 回**（privacy と実装のズレ）→ +数日〜2週間  
2. **申請を 2 回回す**壁時計（待ち ×2 + 文面・ビルドの二重）

## 3. 何が審査を伸ばしやすいか / 伸ばしにくいか

### 伸ばしやすい

- リモート **コード**実行に見える取得（eval / 動的 script）
- 取得物の説明不足（URL のみと書いて metrics を取る、等）
- taught selector 配布が「スクレイピング設定の配布」に見えること
- **権限追加**や privacy の大改訂（今回は raw host 既存なら権限増なしが有利）

### 伸ばしにくい

- サンプル文字バッジを **dist に焼き込み**（ネットワークなし）
- **再取得ボタン**（同じ固定 URL の GET をユーザー操作で再実行）
- paste import（外部通信なし）
- data only・credentials omit・確認ダイアログ後のみ

## 4. 分割 vs 一括

### A. 分割

例: 第1弾 = provider ゼロ + URL のみサンプル + teach 必須 → 第2弾 = スターター（metrics）

| | 内容 |
|--|------|
| 審査 | 第1弾の説明は短く **通る確率は上がりやすい**。第2弾で privacy/listing 更新 → **再レビュー待ち**。合計の壁時計は一括より長くなりがち |
| UX | ストア本命動線（アイコン → 空 popup）が **teach まで遠い**。離脱ポイントそのもの |
| 評価 | 第1弾で「面倒・動かない」が付くと、第2弾後も **低評価が残りやすい** |

### B. 一括（推奨）

例: 空 popup 主 CTA = スターター取込、固定 URL fetch、paste 副経路、既存 teach 非上書き、非保証・Re-teach

| | 内容 |
|--|------|
| 審査 | 説明は少し長いが **1 回のレビュー**で済むことが多い。metrics リモートは文面で「設定ひな型・data only・任意・非保証」を明示 |
| UX | 1〜2 操作でタイル並び → ログイン + ホスト許可 + refresh で数字まで行きやすい |
| 評価 | 初回から「使える」印象を出しやすい。壊れても Re-teach / JSON 差し替え導線があれば「ゴミ拡張」化を抑えられる |

### ざっくり比較

| | 分割 | 一括 |
|--|------|------|
| 初回承認まで | やや短い〜同程度 | 同程度〜説明で +数日 |
| **体験が揃うまで** | 第1弾 + 第2弾 + 再審査 | **初回承認 1 回** |
| ストア評価リスク | 中途半端な第1弾が残る | 初回から本線に近い |
| 離脱 | **高い** | **低い** |

体感:

- 審査の差: **数日〜長くて 2 週間前後**（差し戻しなければ）
- 離脱の差: **インストール直後の 1 分**で決まりやすく、評価 1 件のダメージの方が重い

## 5. このプロダクト固有の前提

- 同梱 provider ゼロ。ユーザーはストアから入ると **まず popup 空状態**
- dogfood で Claude / Codex の teach → 表示は通っている（2026-07-19 recap）
- スターターの目的は **初回到達コストを下げる**こと（selector は腐る前提・ホットフィックスは JSON）
- 言語パック import 等は別件（混同しない）

第1弾を「URL のみ 6 件」に落とすと、承認は取りやすくても **プロダクトとして中途半端**になりやすい。  
分割するなら、第1弾でも少なくとも:

- スターター取込（metrics 付き可）
- 非保証・Re-teach 明記
- privacy / listing と実装の一致

まで入れる。「URL のみ第1弾」は離脱対策として弱い。

## 6. 一括で審査を軽くするチェックリスト

申請前に listing / privacy / submission note と実装を揃える:

1. **送るもの**: なし（GET のみ、credentials omit）
2. **取得物**: 表示名・URL パターン・任意の taught selector/fingerprint（設定ひな型）・任意のサンプル記号アイコン
3. **実行しない**: JSON を code として eval しない（schema 検証のみ）
4. **初回自動 fetch なし**（ボタン確認後のみ）— 必須
5. **公式パートナー設定ではない**・サイト改修で壊れる・`verifiedAt` / Re-teach
6. **サンプルアイコンは自前の文字バッジ**（ベンダー公式ロゴではない）
7. **既存 teach は既定で上書きしない**（上書きは明示確認）
8. host は既存 raw（`ishizakahiroshi/*`）に寄せ、**権限を増やさない**

## 7. アイコン・再取得との関係（補足）

別途の実装議論まとめ（審査遅延の主因ではない）:

| 対象 | 方針メモ |
|------|----------|
| サンプル文字バッジ | ほぼ不変 → **拡張 dist 同梱でもよい**（オフライン・単純） |
| starter.json（URL + metrics） | 腐る → **GitHub 正本**（拡張リリースなしで差し替え） |
| 再取得ボタン | **スターター全体**に付ける価値あり（既存 id スキップ既定・上書きは確認） |

「焼き込み + 初回だけ DL」はアイコンだけで見ると二重管理になりがち。  
**見た目は同梱 or 常にリモートの一方、ひな型はリモート + 再取得**が整理しやすい。
