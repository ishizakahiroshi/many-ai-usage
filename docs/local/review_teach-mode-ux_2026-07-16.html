1. 全体像（まず絵で）

やりたいこと（一言で）

拡張の teach-mode（user がページ上の残枠数字をクリックして「これ追跡して」と教える仕組み）の UX を改修する。目玉は「新規タブで開く（案 D・決定済み）」と「1 タブで複数 metric を連続 teach できる Continuous mode」。本シートで残り 3 判断点を決める。

teach-mode の流れ（改修後の想定）

[Track this element ▸] ↓ 拡張が新規タブで provider の usage ページを開く（案 D） [新規タブ: usage ページ + floating panel] ↓ user が数値をクリック 保存 → panel「保存済み: 1 個」 ↓ 続けてクリック 保存 → panel「保存済み: 2 個」 ↓ [完了して戻る] [options: dashboard に全 metric 反映]

用語ミニ解説（迷ったらここ）

teach-mode
user がページ上の数値を 1 回クリックして「拡張にここを覚えさせる」仕組み。CSS selector と近くのテキストが保存され、次回以降その要素を自動で読む
picker
拡張がページに重ねる「候補ハイライト」+ tooltip。マウスを動かすと候補が変わり、クリックで確定
floating panel
対象タブの右上に固定表示する小さな進行状況ウィンドウ。「保存済み: N 個」を表示し「完了」「キャンセル」ボタンを持つ
metric
追跡する 1 つの数値項目。例:「Codex の週間利用上限 (0% 残り)」「残りのクレジット (0)」
Continuous mode
1 度 picker を起動したら、user が「完了」と言うまで何個でも metric を追加できるモード。逆は「1 回押す = 1 個 teach = 終わり」の 単発 mode
anchor (アンカー)
「教えた要素」を後で探すための情報一式 (CSS selector + 近くのテキスト + タグ構造の fingerprint)
reset anchor
metric の「いつリセットされるか」の日時を書いた要素。例:「リセット: 2026/07/22 20:46」
ヒューリスティック
拡張が「これが数値っぽい」と自動判定するルール群。picker の tooltip 表示にも使う

A. teach-mode の UX 方向性

A1開発 (自分) Q1. Continuous picker mode を採用するか

何を決める？ 1 回 Track this element を押した時に、metric を何個 teach できるか (1 個 or 何個でも)。
なぜ大事？ Codex や Copilot は「週次 + 5-hour + 残クレジット」等 複数 metric を追跡したくなる。1 回 1 個だと Track ボタンを N 回押す + タブが N 個開くので UX が悪い。
たとえると： レシートを 1 枚ずつしか読み取れないスキャナ (単発) と、続けて何枚でも重ねて読み取れるスキャナ (Continuous)。後者の方が家計簿つけるのが楽。
A: Continuous mode の 👍
  • 複数 metric を 1 タブ・1 セッションで登録できる
  • options ↔ usage ページの往復ゼロ
  • 「保存済み: N 個」panel で進捗が可視化される
  • DevTools 系ツール (element inspector) と同じ操作感で自然
A: Continuous mode の 👎
  • 実装コスト高: floating panel の DOM + state 管理 + shadow DOM 化 (対象ページとの衝突回避)
  • panel の DOM が対象ページの z-index / iframe と衝突する可能性 (shadow DOM で対処)
  • 「間違って複数保存」の undo/削除 UI が必要
B: 現状「1 回 1 metric」の 👍
  • 実装がシンプル (現状ロジックをほぼそのまま流用)
  • panel の DOM 衝突リスクなし
  • ユーザーが 1 個だけ追跡したい時にオーバーヘッドなし
B: 現状「1 回 1 metric」の 👎
  • 複数 metric 追跡時に UX が悪い (Track を N 回 + タブが N 個)
  • Codex の "週間 + 残クレジット" のように 2〜3 metric ある provider で毎回ストレス
  • 「これ 1 個 teach したら終わり」の断片的操作感 (継続性がない)

B. Reset anchor（リセット日時）の教え方

B1開発 (自分) Q2. reset anchor をどう取るか

何を決める？ metric ごとに「いつリセットされるか」を、user に明示的に教わるか / 拡張が自動で拾うか。
なぜ大事？ reset 日時があると「あと 4 日で回復」と表示できて便利。ただし teach で毎回聞くと UX が重くなる (1 metric = 2 クリック要求)。
たとえると： 賞味期限。A = レジで毎回「賞味期限どこ？」と店員に聞かれる、B = 商品自体に「2026/07/22」と印刷済みで自動で読み取る。B が普通。
A: 毎回聞く の 👍
  • reset 位置が確実に取れる (自動推測失敗ゼロ)
  • 実装が単純 (picker の状態機械を 2 段階にするだけ)
A: 毎回聞く の 👎
  • teach が 1 metric = 2 クリック必須になり UX 重い
  • reset を持たない metric ("残りクレジット" 数値のみ) では聞かれても Skip 必要で無駄
  • Continuous mode との相性が悪い (連続 teach のリズムが崩れる)
B: 自動推測に任せる の 👍
  • teach 中は「値クリック 1 回」で完結・リズム軽い
  • 「リセット: YYYY/MM/DD」「Resets in N days」形式は高確率で自動抽出可
  • 失敗時のみ後付けで options から reset を re-teach できる (救済経路あり)
B: 自動推測に任せる の 👎
  • 自動抽出のヒューリスティックを実装する必要 (パターンを増やしていく)
  • 「リセット」文言がない provider (例: 「Weekly usage · Fri 09:00 JST」) だと拾えない可能性
  • 誤検出時に user が気づかない可能性 (だが値取得には影響しない)

C. picker の "2026%" 誤検出バグの扱い

C1開発 (自分) Q3. picker が年号を数値候補として拾うバグをどこで直すか

何を決める？ Codex の teach 実験で picker の tooltip が「2026% · click to track」と表示された (日付の "2026" を数値として誤検出)。この修正を本 plan (teach-mode UX 改修) に含めるか、別 bugfix として独立して扱うか。
なぜ大事？ 誤検出は teach の候補プレビューがおかしく見えるだけで、user がクリックすればその要素の selector は正しく保存される (実害小)。ただし tooltip が変な数字を出すのは信頼感を落とす。
たとえると： 家電修理。今回はエアコンの UI 刷新 (teach-mode UX)。ついでにリモコンの液晶にゴミ表示が出るバグ (2026% 誤検出) も直すか、別回で直すか。
A: 本 plan に含める の 👍
  • picker 周りをまとめて触るので、コード側の重複作業が減る
  • UX 改修とヒューリスティック改善が同じ QA で確認できる
A: 本 plan に含める の 👎
  • スコープが膨らむ (本 plan は既に 12 ステップ)
  • リリース時期が bugfix ペースに引きずられる
  • md の主題がぼやける (UX 改修 + heuristic 改善が同居)
B: 別 bugfix md の 👍
  • スコープが明確で md が読みやすい
  • docsweep 上で status を別管理できる (heuristic 改善は別リリースサイクルに乗せられる)
  • 本 plan は UX に集中できる
  • heuristic 改善は将来別のバグ (数値誤認識全般) に拡張しやすい
B: 別 bugfix md の 👎
  • md が増える (docs/local に 1 個追加)
  • 実装時に本 plan と bugfix を両方読む必要 (contract の重複)
