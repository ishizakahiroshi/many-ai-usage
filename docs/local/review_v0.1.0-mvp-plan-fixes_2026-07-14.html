A. 齟齬修正

A1優先度 高 疑い 1: 決定 ID の C1/C2/C3 が plan 章 C1/C2/C3 と表記衝突

何を決める？ plan §C3 章の中で「(C2 決定)」「(C1 決定)」「(C3 決定)」と書かれている決定 ID を、章 ID と混同されない形に書き換えるかどうか。
なぜ大事？ §C3 章の本文で「(C3 決定)」と書くと自己参照に読める。後日の読者(未来の自分)が誤読すると C3 の意味が揺らぐ。

A2優先度 高 疑い 2: page_only が runtime state か snapshot source かで揺れ

何を決める？ plan §C3 と §C7 の「runtime state を page_only に落とす」表現を、synthesis の schema(runtime state.status と snapshot.source)に整合する言い方に直すかどうか。
なぜ大事？ synthesis §3 の runtime state の status enum に page_only は無く、page_only は snapshot の source 値。plan の書き方は「どの schema フィールドを更新するのか」が曖昧で、C2/C3 実装時に迷子の元。

A3優先度 中 疑い 3: ダークモードが MVP スコープ「含む」に明記なし

何を決める？ plan §C6「やること(確定)」に登場する「ダークモード: prefers-color-scheme に追従(実装最小)」を MVP に含めるか、v0.2 に外出しするか。
なぜ大事？ 決定シートで採否した記憶がない項目が UI 章に紛れ込んでいる。スコープ「含む」節にも記載なく、スコープ膨張の疑いあり。

A4優先度 低 疑い 4: C4 完了条件が permission 許可済みを暗黙前提

何を決める？ plan §C4 完了条件「サンプル 2 件の usage ページを開くと popup に自動解析結果が反映される」の書き方を直すかどうか。
なぜ大事？ C4 単体では permission 要求フロー(§C5)がないので実現できない。C4 の完了条件として不整合。

A5優先度 低 疑い 5: v0.1.0 リリース完了条件も permission 許可済み前提

何を決める？ plan 末尾の「完了条件(v0.1.0 リリース定義)」1 行目「サンプル 2 件を登録した状態で、popup に自動解析結果が並ぶ」の書き方を直すかどうか。
なぜ大事？ 疑い 4 と同じ構造。permission 拒否シナリオでは needs_permission カードが並ぶ動線が正なので、リリース完了条件も 2 系統書き分ける方が正確。
