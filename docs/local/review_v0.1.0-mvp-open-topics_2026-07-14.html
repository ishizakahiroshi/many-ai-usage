1. 全体像（まず絵で）

やりたいこと（一言で）

複数 AI サービスのサブスク使用量を「URL を登録するだけ」で 1 画面にまとめて見るブラウザ拡張(MV3・Chrome / Firefox ハイブリッド・サーバーなし)。v0.1.0 は自動解析で拾えなくても tile fallback で一覧化が必ず成立するのを完成定義にする。

plan の構成

C1 リポ骨格 │ ├─ C2 storage schema ─┐ │ │ ├─ C3 自動解析エンジン ─┤ │ │ │ ▼ └─→ C4 取得 pipeline (content + background) │ ▼ ┌───────────────┼───────────────┐ │ │ │ ▼ ▼ ▼ C5 登録 UI C6 dashboard C7 tile fallback │ │ │ └───────┬───────┴───────────────┘ ▼ C8 リリース準備
MVP スコープ (確定済み)： 自動解析 + tile fallback + user visit capture + 手動再取得。教える型と background refresh は v0.2 以降。

用語ミニ解説（迷ったらここ）

bundler
TypeScript のソースを 1 つの JS に束ねて拡張として動く形にする道具。vite / esbuild / wxt など。
manifest
拡張の宣言ファイル(manifest.json)。名前・権限・エントリポイントを書く。MV3 = Manifest V3。
host permission
「このドメインのページを読む許可」。MV3 では optional にでき、ユーザーが URL 登録するタイミングで要求できる。
hydration wait
SPA が JS で本体を描き終わるのを待つこと。document.readyState=complete だけでは不足。
shadow DOM
ページ内に「独立した DOM 領域」を作る web components の仕組み。open なら外から辿れる、closed なら不可。
confidence
自動解析が「これは usage の値だ」と思う自信度(0-1)。閾値以下なら採用しない。
tile fallback
数値を抽出できない URL を、「開くリンク付きの表示型カード」として並べて一覧化価値を残す最終手段。
runtime validator
storage から読んだ JSON を実行時に型チェックする道具。zod / valibot など。

3. これから決めること（本日の相談）

各項目で選択肢を選び、メモ欄に補足を書いてください。最後の「決定サマリを出力」で選んだ内容がまとまります。

A. C1 リポ骨格

A1 bundler の選定 (§C1-1)

何を決める？ TypeScript ソースから Chrome/Firefox 用の拡張成果物を作る道具を何にするか。
なぜ大事？ 選択次第で manifest 差分の吸収層(§Cross-2)・UI framework(§C6-1)・CI 構成が連動する。ここが要石。

A2 LICENSE (§C1-2)

何を決める？ リポの LICENSE を何にするか。
なぜ大事？ 個人 OSS の慣例で MIT 想定だが、初期に決めておく方が後から書き換えるより健全。

B. C2 storage schema

B1 runtime validator (§C2-1)

何を決める？ storage から読んだ JSON を実行時に型チェックする道具。
なぜ大事？ 拡張は bundle size に敏感(popup 起動速度)。ただし schema 数は少ないので数 KB 差。

B2 metrics 配列の MVP 範囲 (§C2-2)

何を決める？ v0.1.0 では metrics 配列の taught 分岐を実装するか、型だけ用意して writer は書かないか。
なぜ大事？ 教える型は v0.2 送りが確定済み。ここは型だけ入れておくと v0.2 の migration が要らない。

B3 stale 判定閾値 (§C2-3)

何を決める？ 最終取得から何経過で「古い(stale)」表示にするか。
なぜ大事？ 短すぎると常に stale で信頼を落とし、長すぎると古い値を成功値のように見せる。

C. C3 自動解析エンジン

C1 多言語ラベル辞書の置き場所 (§C3-1)

何を決める？ 「remaining」「残り」「剩余」等の label 辞書をどこに持つか。
なぜ大事？ detector が参照するので host に近い方が読みやすいが、UI とも共有する可能性がある。

C2 shadow DOM 対応範囲 (§C3-2)

何を決める？ どこまで DOM を歩いて数値候補を探すか。
なぜ大事？ 広げるほど拾えるが、closed shadow root と cross-origin iframe は原理的に無理。

C3 confidence 採用閾値と tile fallback (§C3-3)

何を決める？ confidence 何以上を採用するか、下回った時 v0.1.0 では何にフォールスルーするか。
なぜ大事？ 教える型が v0.2 送りなので、低 confidence 時の受け皿は tile fallback しかない。

D. C4 取得 pipeline

D1 hydration wait (§C4-1)

何を決める？ SPA が描き終わるのを何秒待つか、どう待つか。
なぜ大事？ 短いと空の skeleton を読む、長いと popup 描画が遅くなる。

D2 手動再取得中に tab が閉じた時の handling (§C4-2)

何を決める？ 再取得中に対象タブが閉じられたら runtime state をどうするか。
なぜ大事？ silently 成功偽装は避ける。ユーザーに「なぜ更新されなかった」を残す。

E. C5 登録 UI + permission

E1 needs_permission の扱い (§C5-1)

何を決める？ host permission 拒否時の状態を synthesis §3 の status enum に足すか、runtime state 側で扱うか。
なぜ大事？ schema v1 を凍結するので、後から enum を足すと migration が発生する。

E2 サンプル 2 件の URL 具体 (§C5-2)（記入式）

お願い： Claude と Codex(OpenAI) の usage ページ URL を確定してメモ欄に書く。実 URL は公開情報なので機密ではない(実 usage 値・アカウント ID は書かない)。

F. C6 ダッシュボード UI

F1 UI フレームワーク (§C6-1)

何を決める？ popup / options / full page を書く道具。
なぜ大事？ bundler(§A1) と連動。popup 起動速度に直結する bundle size を左右する。

F2 popup の情報密度 (§C6-2)

何を決める？ 1 プロバイダあたり popup にいくつの metric を出すか。
なぜ大事？ 複数 window(5h / weekly / monthly) を持つプロバイダで隠しすぎると synthesis §6 の「複数 window を隠さない」に反する。

F3 icon 生成タイミング (§C6-3)

何を決める？ make-icon skill で icon 一式を作るのを C8(リリース準備) で扱うか、もっと早いか。
なぜ大事？ 開発中の toolbar icon はプレースホルダで足りる。ストア素材で必須。

G. C7 tile fallback

G1 tile 化の条件 (§C7-1)

何を決める？ どのタイミングで自動解析を諦めて tile fallback にするか。
なぜ大事？ 候補ゼロだけを条件にすると誤検出が数字として出る、低 confidence も含めると tile 化が増える。

G2 tile に favicon を出すか (§C7-2)

何を決める？ tile カードにサイトの favicon を表示するか。
なぜ大事？ favicon 取得には host permission か Google の favicon proxy が要る。プライバシー原則(外部送信しない)と噛む。

H. C8 リリース準備

H1 version 戦略 (§C8-1)

何を決める？ 初回リリースを 0.1.0 で出すか、0.0.1 で pipeline を先に通すか。
なぜ大事？ ストア審査を通した実績がある方が本番の 0.1.0 が確実。

H2 privacy policy の置き場所 (§C8-2)

何を決める？ ストア審査に必須の privacy policy をどこに置くか。
なぜ大事？ Chrome Web Store は URL 提出必須。README 節でも技術的には可だが GitHub Pages の方が住所が安定。

I. 横断論点

I1 secrets-scan 実配線の動作確認タイミング (§Cross-1)

何を決める？ .githooks/ 配線の動作確認をいつやるか。
なぜ大事？ 実装が進んでから壊れていることに気付くと手戻りが痛い。

I2 Chrome/Firefox manifest 差分の吸収層 (§Cross-2)

何を決める？ manifest.json の Chrome/Firefox 差分を build 時に生成するか、実行時分岐で吸収するか。
なぜ大事？ 実行時分岐だと manifest 側の diff は最小だが、build 時生成の方が両ストア向けに素直。

I3 bundler / UI framework の確定順序 (§Cross-3)

何を決める？ §A1(bundler) と §F1(UI framework) をどの順序で決めるか。
なぜ大事？ wxt を選ぶと framework 選択も影響する等、連動している。

I4 host permission 拒否後の provider entry の掃除 (§Cross-4)

何を決める？ 拒否したまま残された provider entry を放置するか、UI で明示的に削除を促すか。
なぜ大事？ 放置すると popup が「permission 拒否のカード」で埋まる。
