4. 齟齬の疑い(5 件)

plan を読み返して洗った疑い。断定ではなく「読者が誤読しうる/矛盾しうる」箇所として列挙。

疑い 1(修正推奨・表記衝突)

決定 ID の C1/C2/C3 が plan 章の C1/C2/C3 と表記衝突

C3 章の本文に「(C2 決定)」「(C1 決定)」「(C3 決定・§C7 と噛み合わせる)」と書いてあるが、これは決定シートの群 C の連番(shadow DOM / i18n / confidence)を指しており、plan 章の C1/C2/C3 とは無関係。plan の C3 章の中で「(C3 決定)」と書くと自己参照に読める。

該当箇所: plan §C3 「やること(確定)」の 3 箇条。

推奨修正: 決定 ID を C-1 / C-2 / C-3 のようにハイフン区切りに変えるか、群 C を別文字(例: J1/J2/J3)にリネームして章とぶつからないようにする。本レビューシートでは仮に C-1 表記を使用している。

疑い 2(用語混同の可能性)

page_only の帰属先が runtime state か snapshot source かで揺れている

plan §C3 と §C7 に「runtime state を page_only に落とす」とあるが、synthesis §3 の ProviderRuntimeState には pageBinding: unbound | bound | stale しかなく、page_only は synthesis §4 の NormalizedSnapshot.source: dom | user_taught | page_only の値。

該当箇所: plan §C3 「DOM 対応範囲」節、plan §C7 「tile 化の条件」節。

推奨修正: 「runtime state を page_only に落とす」→「snapshot の source を page_only にし、runtime state の status を ok(表示型) にする」等、schema の 2 概念(source と status)を明確に分ける表現に。

疑い 3(スコープ膨張の可能性)

ダークモード対応が MVP スコープの「含む」に明記なし

plan §C6 の「やること(確定)」に「ダークモード: prefers-color-scheme に追従(実装最小)」と書かれているが、スコープ節の「含む」箇条には言及なし。決定シートにも登場しない。決定を経ずに UI 章に紛れ込んだ可能性。

該当箇所: plan §C6 やること節の最終行。

推奨判断: (a) MVP に入れるならスコープ「含む」に 1 行追加、(b) 入れないなら §C6 から削除して v0.2 ロードマップへ移す。個人的には(b)がスコープ膨張抑止として推奨(実装最小と書いても popup UI の CSS 分岐は増える)。

疑い 4(前提の書き漏れ)

C4 完了条件が「permission 許可済み」を暗黙前提にしている

plan §C4 の完了条件「サンプル 2 件の usage ページを開くと popup に自動解析結果が反映される」は、C5 の「サンプル 2 件は初回インストール時に登録済み(permission は未許可)」と読み合わせると、permission 許可という前段ステップを暗黙にしている。C4 単体では実現できない。

該当箇所: plan §C4 完了条件 1 行目。

推奨修正: 「permission 許可済みのサンプルのページを…」に前提を明記する、あるいは C4 の完了条件を「permission 許可済み host で content script が snapshot を書き込める」まで抽象化して、「popup に反映」は C6 完了条件に持たせる。責務分離的には後者が綺麗。

疑い 5(前提の書き漏れ・疑い 4 の派生)

v0.1.0 リリース完了条件も「permission 許可済み」を暗黙前提

「完了条件(v0.1.0 リリース定義)」の 1 行目「サンプル 2 件を登録した状態で、Chrome/Firefox どちらでも popup に自動解析結果が並ぶ」も疑い 4 と同じ構造。permission 拒否の場合は needs_permission カードが並ぶ動線が正なので、完了条件も許可済みシナリオに絞って書いている。

推奨修正: 「(permission 許可済みシナリオで) 自動解析結果が並ぶ / 拒否シナリオでは needs_permission カードが並ぶ」の 2 系統を並記する。

5. 疑い以外の観察(参考)

  • アーキ図の「MutationObserver + 短い hydration wait」の数値詳細は本文 §C4 の D1 決定側にある。図には出さず本文で補足する現形は問題なし
  • C1 完了条件の「popup が空の状態で開く」← C5 実装前のため空でよいが、読者が「C5 のサンプル 2 件は?」と迷う可能性は微(C1 は骨格のみと明記されているため問題なし)
  • C8 リスク節「Claude/Codex の usage ページ URL 変更を changelog-freshness で拾えない」← 「拾えない」と明示している(問題なし)
  • B3 メモの反映が §C2 に UI 責務の記述(「手動再取得ボタン(§C6)が担う」)として現れるやや不自然が齟齬ではない。§C6 に集約して §C2 は stale 定義のみにする軽微な整理は可能
  • F3 前倒しで C1 と C8 に icon が二度登場するが、「初期セット / 最終素材化」で責務分離済み(問題なし)

6. 目視チェック後の推奨アクション

優先度 高: 疑い 1(表記衝突) / 疑い 2(page_only の帰属)

優先度 中: 疑い 3(ダークモード スコープ)

優先度 低: 疑い 4 / 5(前提の書き漏れ・完了条件の許可済み/拒否 2 系統化)

いずれも実装着手前に潰しておくと C1 以降の作業中の解釈揺れがなくなる。修正は plan md への直接反映を推奨。
