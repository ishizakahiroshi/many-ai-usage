## 背景と目的

**many-ai-usage**（`C:\dev\github\public\many-ai-usage`）は、複数 AI サービスのサブスク使用量（レート上限・残枠）を 1 画面に一覧表示するブラウザ拡張（MV3・Chrome / Firefox ハイブリッド・サーバーなし・ブラウザ内完結・OSS）を新規開発するプロジェクト。プロジェクト概要は同リポの `CLAUDE.md` を先に読むこと。

設計の入口思想（重要・解析の評価軸）:

- **プリセットのプロバイダ定義を同梱しない**。ユーザーが自分で AI プロバイダ（usage ページの URL 等）を登録する。同梱するのは Claude / Codex のサンプル 2 件のみ（サンプルであってハードコードではない）
- 取得モードは 3 段の梯子（上から順に試す）:
  1. **自動解析（第一・既定）**: 登録 URL のページをブラウザ内ヒューリスティックで自動解析し、使用量らしきもの（%表記・progress/meter 要素・「リセット/reset」近傍の日時。日英中対応）を抽出して自動でゲージ化。プロバイダ知識ゼロ・メンテ不要が狙い
  2. **ユーザー補正（フォールバック）**: 自動解析が拾えない/誤検出した時だけ、ユーザーが要素クリックで教える・不要メトリクスを消す
  3. **そのまま表示型（最終フォールバック）**: ページをタイルとして埋め込み表示
- **自動解析はローカルヒューリスティックのみ。ページ HTML を外部 AI に送らない**（プライバシー原則・ストア審査の生命線）
- 運営側はプロバイダ知識をメンテしない。**UI/UX に専念**する
- 認証はユーザーのログイン済みブラウザセッションを読むだけ。外部送信なし
- ダッシュボードは自動更新（ユーザーがページを開いていない時にどう再読取するかは設計上の急所 → C3 で検討）

この構想を固めるため、方向性が近い既存 OSS 2 件の内部構造を解析し、設計に使える知見と避けるべき落とし穴をレポート化する。

## C1: AI_Usage_Dashboard の解析

### 解析観点

1. **データ取得方式**: 各プロバイダ（Codex / Cursor / Claude Code / Gemini）の usage をどう取っているか（ログイン済み Cookie での内部 API fetch / DOM スクレイプ / ローカルファイル読取の別）。エンドポイントとレスポンスの扱い方
2. **custom HTTP/HTTPS JSON sources の実装**: ユーザー定義ソースのスキーマ・登録 UI・バリデーション・エラーハンドリング。「ユーザーが任意プロバイダを追加する」体験としてどこまでできているか / 何ができないか
3. **MV3 permissions 設計**: host_permissions / optional_host_permissions の使い分け、プロバイダ有効化時の権限要求フロー
4. **Firefox 対応**: `dist/firefox/` 等のビルド差分の作り方（manifest 差分・API 吸収・ビルドスクリプト）。どこまで動いていてどこが未完か
5. **UI 構成**: ダッシュボードの画面構成（ポップアップ / フルページ / バッジ）、使用率の見せ方、リセット時刻表示、複数プロバイダの並べ方
6. **更新系**: ポーリング間隔・キャッシュ・レート制限対策（429 回避）・セッション切れ検出と再ログイン導線
7. **弱点・不満点**: UX 観点で「ユーザーが嬉しくない」箇所（many-ai-usage が改善で勝てる点）

## C2: openusage の解析

### 解析観点

1. **プラグイン駆動アーキテクチャ**: プラグイン定義のスキーマ（何を宣言すると 1 プロバイダが増えるか）・ロード機構・プラグインの配布/更新方法
2. **プロバイダ別の取得方式一覧**: 各プラグインが認証情報をどこから取り（ローカルファイル / Cookie / API キー）、どのエンドポイントをどう叩いているか。ブラウザ拡張に移植できる方式とできない方式（ネイティブアプリ専用）の仕分け
3. **正規化モデル**: プロバイダごとにバラバラな usage 概念（5h 窓 / 週次 / クレジット / リクエスト数）を、どういう共通データ構造に正規化して UI に流しているか
4. **更新系・エラー処理**: ポーリング設計、失敗時のユーザー表示
5. **UI**: メニューバー常駐 UI での情報密度の作り方（ブラウザ拡張のポップアップ/バッジに翻訳できる知見）
6. **弱点・不満点**: 同上

## C3: 統合 — many-ai-usage 設計への示唆

### 作業内容

C1 / C2 のレポートを踏まえ、以下を短くまとめる（新規調査はしない）:

1. **自動解析 + 教える型の実現性**: 2 つの OSS の取得方式と比較して、「ページのヒューリスティック自動解析（第一）+ ユーザーが数字をクリックして補正（フォールバック）」方式の強み・弱み・実装上の急所（SPA の再レンダリングでセレクタが壊れる問題、描画完了の待ち方、%以外の表記ゆれへの対策候補など）
   - あわせて **自動更新の機構**を検討: ユーザーが usage ページを開いていない時にどう再読取するか（非アクティブタブ / 拡張ページ内 iframe + ヘッダ加工 / ユーザー訪問時キャプチャ、の各候補の実現性とストア審査リスク）
2. **レシピ/プロバイダ登録のスキーマ案**: openusage のプラグインスキーマと AI_Usage_Dashboard の custom sources を参考に、many-ai-usage のユーザー登録データ構造（URL・取得モード・教えた要素・表示名・更新間隔）の叩き台
3. **usage の共通正規化モデル案**: パーセント型 / 残数型 / リセット時刻の共通表現
4. **避けるべき落とし穴リスト**: 429・セッション切れ・SPA 描画待ち・ストア審査（権限の説明責任）等、2 OSS から学んだもの
5. **UX で勝つポイント**: 既存 OSS の不満点から導く「ユーザーが嬉しいデザイン」の要件リスト（ここが many-ai-usage の主戦場。UX 最優先）
